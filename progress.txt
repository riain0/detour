## Codebase Patterns
- Rust workspace; quality gate = `cargo check --workspace` + `cargo test --workspace` (no web UI, no typecheck/lint step).
- Proto lives in crates/proto/proto/detour.proto; build.rs codegen via tonic_prost_build. After editing, generated types are at `detour_proto::detour::*` (re-exported as `detour_proto::*`). Module-style oneof enums: `detour_proto::detour::agent_message::Payload::*`, `broker_message::Payload::*`, etc.
- ADDING a field/variant to a proto oneof is NOT free: every exhaustive `match msg.payload` breaks. Known match sites: broker/src/relay.rs (open_tunnel on agent_message), agent/src/tunnel.rs (on broker_message). Add a `Some(...::Raw(_)) => {}` arm.
- ADDING an RPC to the `service Detour` forces broker/src/relay.rs `impl Detour for RelayService` to implement it (assoc stream type + method). Stub with `Err(Status::unimplemented(..))` if deferring real logic.
- Existing data plane: sidecar↔broker = RelayRequest RPC (HTTP-shaped, request_id keyed). agent↔broker = OpenTunnel bidi (BrokerMessage→agent carries RelayRequest; AgentMessage→broker carries RelayResponse). OutboundTunnel = raw TCP egress (agent→broker→cloud), good reference for raw byte pumping (read loop + data/fin oneof, 16384 buf).
- Connection bookkeeping: broker/src/connections.rs has ConnectionMap (session→agent tx) + PendingRequests (request_id→resp tx). Sessions keyed by detour_core::SessionId.
- IMPORTANT prior state: feature branch started from ~1600-line uncommitted working tree on main (CLI run/env/auth, expanded layer, broker auth). Committed as baseline `a06ce8a` BEFORE US-001 so per-story commits stay isolated. Do NOT assume a clean bootstrap.

## 2026-06-01 - US-002
- Wired the broker-side raw byte-stream data plane: RelayConnection now does real per-connection multiplexing.
- Files changed:
  - crates/broker/src/connections.rs: new `RawConnTx` alias + `RawConnections` map (connection_id -> sidecar resp tx). `register`/`remove`/`deliver`. `deliver` drops the entry on is_eof or when the sidecar stream is gone. +2 unit tests.
  - crates/broker/src/relay.rs: implemented `relay_connection` (was unimplemented stub). Opening frame -> (session_id, connection_id) routing; resolves agent tunnel via ConnectionMap; forwards frames to agent as `BrokerMessage::Raw` in order; is_eof on sidecar half breaks the pump but leaves the routing entry for the agent half. Filled the `agent_message::Payload::Raw(frame)` arm in open_tunnel to `raw_connections.deliver(frame)`. Added `raw_connections` field to RelayService.
  - crates/broker/src/main.rs: construct `RawConnections::default()`.
- Quality: `cargo check --workspace` clean; `cargo test --workspace` = 15 pass (was 13).
- **Learnings for future iterations:**
  - RawConnections is the raw-plane analogue of PendingRequests: PendingRequests is keyed by request_id (HTTP plane), RawConnections by connection_id (raw plane). Both deliver agent frames arriving on the single OpenTunnel AgentMessage stream back to the originating sidecar RPC.
  - Close semantics are split-half: sidecar eof breaks the broker's read loop but the routing entry survives so the agent can still flush its half; the entry is reaped when the agent sends is_eof (deliver drops it) OR the sidecar stream dies without eof. A connection where neither side sends is_eof can leak an entry — acceptable for US-002, tighten in US-012 robustness if needed.
  - US-003 (agent forwarder) must: on `broker_message::Payload::Raw` opening frame, dial upstream per connection_id and pump bytes both ways, emitting `AgentMessage::Raw` frames back with the same connection_id; is_eof closes the matching half. The broker is now ready to route those frames.
  - US-004 (sidecar) is the client of relay_connection: it must set connection_id (broker rejects empty) and service_name on the opening frame.

## 2026-06-02 - US-003
- Wired the agent-side raw byte-stream forwarder: per-connection dial + bidirectional pump.
- Files changed:
  - crates/agent/src/forwarder.rs: new `forward_connection(open, local_port, tx, broker_rx)`. Dials 127.0.0.1:local_port, writes the opening frame's initial payload, then runs two halves: a spawned upstream->broker reader emitting `AgentMessage::Raw` frames (same connection_id, 16384 buf, eof on read 0/err) and a broker->upstream loop draining `broker_rx`. `raw_frame()` helper builds the AgentMessage. +2 unit tests (echo pump + dial-failure-emits-eof).
  - crates/agent/src/tunnel.rs: filled `broker_message::Payload::Raw(frame)` arm. inflight_connections map (connection_id -> mpsc::Sender<RawConnFrame>). Opening frame: route_map lookup -> spawn forward_connection with a fresh channel (insert unless is_eof). Continuation frames: forward to the per-connection channel; drop entry on eof or send failure. Unknown service_name -> warn + drop.
- Quality: `cargo check --workspace` clean; `cargo test --workspace` = 17 pass (was 15).
- **Learnings for future iterations:**
  - DEADLOCK FIXED: original draft always ran the broker->upstream `while broker_rx.recv()` loop even when the opening frame had is_eof=true. With the client half already closed there are no more client frames, so recv() blocks until the sidecar drops the RelayConnection stream. The forward_connection task never returns and any handle.await on it hangs (manifested as the echo test running >60s). Fix: branch on open.is_eof — if set, just shutdown the upstream write half and skip the pump. The upstream->broker reader still drains the response. Lesson: a raw-pump half must terminate on the FIRST eof signal it sees (opening frame OR a later frame), never assume more frames will arrive to unblock recv().
  - Test gotcha: `let (_broker_tx, rx)` keeps the sender alive to end of scope (single underscore != immediate drop), so any code awaiting the receiver's close hangs. Use the is_eof opening-frame path or `drop(broker_tx)` explicitly in tests.
  - US-004 (sidecar) is the producer side: it must set connection_id + service_name on the opening RawConnFrame and call broker `relay_connection`. The broker (US-002) routes to this agent forwarder, now ready.
  - Hung test binaries survive `pkill -f "cargo test"` (cargo parent dies, detached test binary keeps running). Kill with `pkill -9 -f target/debug/deps/<crate>`. macOS has no `timeout`; guard test hangs by running a single -p crate in the background and polling.
